# Family Office Brain — Мастер разбора кейса
## Техническая спецификация для разработки в Cursor (MVP)

Версия: 1.0. Основана на decision-engine.md v0.3 (7 кейсов, 5 принципов) и seed-data.json.

---

## 1. Что строим и для кого

Личный инструмент одного пользователя (принципала/руководителя FO) для разбора реальных и гипотетических кейсов по методологии Decision Engine, накопленной за первые 7 кейсов. Не CRM, не база знаний в широком смысле — рабочий инструмент мышления, который со временем накапливает и уточняет принципы принятия решений.

**MVP explicitly НЕ включает:** многопользовательский доступ, права для сотрудников FO, клиентский интерфейс, мобильное приложение. Это всё — возможные следующие этапы, не часть первой версии.

**Единственная функция MVP: Мастер разбора кейса.** Пользователь вводит новый кейс → система ведёт его по релевантной ветке Decision Engine, подсказывает вопросы из библиотеки, показывает связанные существующие принципы/паттерны → пользователь фиксирует решение и логику → система предлагает, что это подтверждает/уточняет/добавляет в базу, пользователь утверждает или правит → кейс сохраняется, база обновляется.

Ключевое ограничение по духу проекта: **система никогда не принимает решение о статусе принципа сама** — она только предлагает (кандидат → подтверждён N-й раз → принцип), пользователь всегда подтверждает вручную. Это прямое отражение принципа p_core из самой базы («не решай за принципала») — программный продукт должен следовать той же логике, которую описывает.

## 2. Модель данных

Сущности (см. точные поля в `seed-data.json`, который используется как данные первого запуска):

**Case** — кейс. `id, title, domain, status (hypothetical | real_in_progress | real_closed), branchId, facts (text), questionsAsked (Question[]), decisionTree (text/markdown), outcome (text), createdAt, updatedAt`. Связан many-to-many с Principle и Pattern через таблицу подтверждений (`CaseConfirmsPrinciple`, `CaseConfirmsPattern`) — так и накапливается счётчик подтверждений.

**DecisionEngineBranch** — ветка рамки (сейчас три: спор/ответственность, административно-комплаенсное требование, структурное/корпоративное решение). `id, name, appliesWhen (text), blocks (string[], упорядоченный список вопросов блока)`.

**Question** (Decision Questions library) — `id, text, domainTags (string[]), sourceCaseIds (string[])`. Пополняется вручную и предложениями ИИ по итогам разбора кейса.

**Principle** — `id, title, statement, status (candidate | pattern | principle | explicit_stated), confirmedByCaseIds (string[]), personalCalibrationNote (text, optional)`. Статус повышается автоматически-предложением при достижении порога (по методу проекта — 3+ разнородных кейса для перехода pattern → principle), но применяется только по подтверждению пользователя.

**Pattern** — то же самое, но для более узких/тактических находок, которые не обязаны становиться принципами (`status: single_case | two_case | tactical | candidate`).

**OpenQuestion** — `id, text, relatedCaseId (optional), status (open | resolved), resolutionNote (optional)`.

Все сущности — плоские, без сложной иерархии. Это сознательно просто: продукт растёт вместе с накоплением кейсов, переусложнять схему на старте не нужно.

## 3. Технологический стек (рекомендация)

Учитывая: один пользователь, локальное использование, разработка через Cursor с ИИ-ассистентом, важна скорость сборки MVP, а не масштабируемость.

- **Next.js (App Router) + TypeScript** — фронтенд и API-роуты в одном проекте, хорошо поддерживается Cursor.
- **SQLite + Prisma ORM** — файловая БД, ноль конфигурации инфраструктуры, полностью локально. При первом запуске приложение импортирует `seed-data.json` (скрипт `prisma/seed.ts`).
- **Tailwind CSS** — быстрая вёрстка без лишних решений по дизайну на старте.
- **Anthropic API (Claude)** для ИИ-подсказок внутри мастера (см. раздел 5) — серверный вызов из API-роута, ключ хранится в `.env.local`, никогда не уходит на клиент.
- Запуск: `npm run dev`, локально на `localhost:3000`. Деплой в облако — не требуется для MVP, но Next.js это позволяет сделать позже без переписывания.

Альтернатива, если хочется собрать ещё быстрее и проще (ценой менее гибкого UI): **Python + Streamlit** поверх SQLite напрямую. Годится, если приоритет — минимальное время до первого рабочего прототипа, а не полировка интерфейса. Рекомендация всё же Next.js, потому что мастер кейса — это по сути многошаговая форма с состоянием, для которой React подходит лучше, чем Streamlit-перерисовка страницы на каждом шаге.

## 4. Экраны

**4.1 Дашборд.** Список кейсов (фильтр по domain/status), счётчик принципов по статусам, список открытых вопросов. Точка входа — кнопка «Новый кейс».

**4.2 Мастер нового кейса** (ядро продукта), шаги:

1. **Факты.** Свободный текст — что произошло. (Пользователь может просто вставить сырой текст, как формулирует сам, без структурирования — система структурирует дальше.)
2. **Классификация.** Система (через ИИ, см. раздел 5) предлагает домен и ветку Decision Engine (или пользователь выбирает вручную/переопределяет). Если ни одна ветка не подходит — можно продолжить без ветки (не насиловать рамку под кейс, который в неё не укладывается, как было с кейсами 2/7/7b).
3. **Вопросы.** Показываются блоки выбранной ветки + релевантные вопросы из библиотеки (по domainTags и по семантическому совпадению через ИИ-подсказку). Пользователь отвечает по каждому или помечает «неприменимо». Может добавить свой вопрос — он уходит в библиотеку с пометкой источника (этот кейс).
4. **Дерево решений / рассуждение.** Свободный текст или простой блочный редактор («если… → тогда…»). Не обязателен к строгой форме — если кейс не укладывается в дерево, можно оставить как связный текст.
5. **Итог.** Что решили, что произошло по факту (для real_in_progress можно оставить пустым и вернуться позже).
6. **Что это даёт базе.** Система (ИИ) предлагает список: «это может подтверждать принцип X», «это может быть новым паттерном Y», «это открывает вопрос Z». Пользователь по каждому пункту: подтвердить / отклонить / отредактировать формулировку. Ничего не сохраняется в базу принципов без явного подтверждения на этом шаге.
7. **Сохранить.** Кейс уходит в журнал, подтверждённые принципы/паттерны обновляют счётчики и (если достигнут порог) предлагают повышение статуса — снова с подтверждением пользователя, не автоматически.

**4.3 Библиотека принципов и паттернов.** Просмотр/поиск, фильтр по статусу и домену, у каждого — список кейсов-подтверждений со ссылками.

**4.4 Открытые вопросы.** Список, с возможностью связать ответ с новым или существующим кейсом и закрыть.

**4.5 Экспорт.** Кнопка «Собрать decision-engine.md» — генерирует markdown в том же формате, что нынешний живой документ (для чтения человеком, шаринга, бэкапа вне БД).

## 5. Интеграция ИИ

Используется на трёх шагах мастера: классификация домена/ветки (4.2.2), подсказка релевантных вопросов и принципов (4.2.3), предложение по обновлению базы (4.2.6). Везде — предложение, не решение; финальное слово всегда за пользователем на экране.

Вызов — серверный (Next.js API route), модель — Claude, запрос собирается так: в системный промпт зашивается сокращённая версия методологии проекта (раздел 0 текущего decision-engine.md — цель, правило метода про 3+ кейса) плюс сериализованный список текущих Principle/Pattern/Question (из БД, не статично — обновляется с каждым новым кейсом). Пользовательское сообщение — факты нового кейса. Ответ форсируется в структурированный JSON через tool use/structured output (схема ниже), не свободный текст — чтобы UI мог напрямую рендерить предложения как список с чекбоксами.

Примерная схема ответа:
```json
{
  "suggestedDomain": "string",
  "suggestedBranchId": "string | null",
  "relevantQuestionIds": ["string"],
  "newQuestionSuggestions": ["string"],
  "relevantPrincipleIds": ["string"],
  "relevantPatternIds": ["string"],
  "updateProposals": [
    {"type": "confirm_principle | confirm_pattern | new_pattern | new_open_question",
     "targetId": "string | null",
     "title": "string",
     "rationale": "string"}
  ]
}
```

Это прямое повторение того, что делалось вручную в этом чате при разборе каждого кейса — система должна имитировать тот же ход рассуждения: искать первопричину, проверять применимость существующих веток, предлагать, что подтверждается или уточняется, без домысливания за пользователя.

## 6. Порядок сборки в Cursor (предлагаемые итерации)

1. Скаффолд Next.js + Prisma + SQLite, схема БД по разделу 2, скрипт импорта `seed-data.json`.
2. Дашборд + список кейсов (без мастера) — чтобы сразу видеть, что данные из текущей базы (7 кейсов) корректно живут в новой структуре.
3. Мастер нового кейса — шаги 1, 2, 4, 5 без ИИ (голая форма, ручной выбор ветки/вопросов).
4. Библиотека принципов/паттернов + открытые вопросы — экраны просмотра.
5. Интеграция Claude API — подсказки на шагах 2, 3, 6 мастера.
6. Экспорт в markdown.

Каждый пункт — самостоятельно тестируемый прирост; можно останавливаться после любого и уже пользоваться инструментом.
