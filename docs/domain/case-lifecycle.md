# Case Lifecycle

Status: Draft v1

## Purpose

This document defines the lifecycle of a Case as a domain state machine.

The lifecycle controls:
- which actions are allowed;
- which transitions are valid;
- which domain events are emitted;
- which invariants must hold.

## States

### `draft`

The Case exists but is not yet ready for analysis.

Allowed actions:
- add or update Objective;
- add Facts, Assumptions, Unknowns, and Risks;
- request activation.

Exit condition:
- the Case has a defined Objective;
- the Case has sufficient initial context for analysis.

---

### `active`

The Case is open and under analysis.

Allowed actions:
- add evidence;
- confirm or reject Facts;
- add or revise Assumptions;
- add or resolve Unknowns;
- add or update Risks;
- generate or revise Options;
- create or revise a Recommendation;
- request a Principal Decision;
- pause the Case;
- close the Case.

Exit conditions:
- move to `awaiting_decision` when a Principal Decision is required;
- move to `paused` when work is intentionally suspended;
- move to `closed` when the Outcome is achieved or the Case is terminated.

---

### `awaiting_decision`

The Case has an active Recommendation and requires an authorised human decision.

Allowed actions:
- record a Principal Decision;
- revise the Recommendation if material context changes;
- return to active analysis;
- pause the Case;
- close the Case.

Exit conditions:
- move to `active` when the Decision requires further work or execution planning;
- move to `paused` when the Case is suspended;
- move to `closed` when no further work is required.

---

### `paused`

The Case is temporarily inactive.

Allowed actions:
- add new evidence;
- update context;
- resume the Case;
- close the Case.

Exit conditions:
- move to `active` when work resumes;
- move to `closed` when the Case is terminated.

---

### `closed`

The Case is complete and immutable except for administrative metadata.

Allowed actions:
- add closure metadata;
- reopen the Case only through an explicit reopen command.

Exit condition:
- move to `active` only through `ReopenCase`.

## Allowed Transitions

| From | Command | To | Event |
|---|---|---|---|
| `draft` | `ActivateCase` | `active` | `CaseActivated` |
| `active` | `RequestPrincipalDecision` | `awaiting_decision` | `PrincipalDecisionRequested` |
| `active` | `PauseCase` | `paused` | `CasePaused` |
| `active` | `CloseCase` | `closed` | `CaseClosed` |
| `awaiting_decision` | `RecordPrincipalDecision` | `active` | `PrincipalDecisionRecorded` |
| `awaiting_decision` | `ReturnToAnalysis` | `active` | `CaseReturnedToAnalysis` |
| `awaiting_decision` | `PauseCase` | `paused` | `CasePaused` |
| `awaiting_decision` | `CloseCase` | `closed` | `CaseClosed` |
| `paused` | `ResumeCase` | `active` | `CaseResumed` |
| `paused` | `CloseCase` | `closed` | `CaseClosed` |
| `closed` | `ReopenCase` | `active` | `CaseReopened` |

## Transition Rules

### Activate Case

`ActivateCase` is allowed only when:
- Objective is present;
- the Case is in `draft`;
- required metadata is valid.

### Request Principal Decision

`RequestPrincipalDecision` is allowed only when:
- the Case is `active`;
- there is exactly one active Recommendation;
- the Recommendation has rationale;
- material Risks and Unknowns are visible;
- the requested Decision is stated explicitly.

### Record Principal Decision

`RecordPrincipalDecision` is allowed only when:
- the Case is `awaiting_decision`;
- the Decision is made by an authorised human;
- the Decision references the active Recommendation or explicitly rejects it.

The recorded Decision must include:
- decision;
- decision-maker;
- timestamp;
- rationale, if provided;
- conditions or constraints, if any.

### Close Case

`CloseCase` is allowed when:
- the intended Outcome has been achieved; or
- the Case has been explicitly terminated; or
- no further action is required.

Closure must include:
- closure reason;
- final Outcome status;
- unresolved Risks, if any;
- follow-up obligations, if any.

### Reopen Case

`ReopenCase` is allowed only when:
- the Case is `closed`;
- new material information or a new obligation exists;
- the reopen reason is recorded.

Reopening does not remove or rewrite the previous closure history.

## Invariants

The Case lifecycle must always preserve the following:

1. A Case has exactly one current lifecycle state.
2. State changes occur only through explicit Commands.
3. Every successful state change emits exactly one lifecycle Event.
4. Historical Events are immutable.
5. A closed Case cannot be modified unless it is reopened.
6. A Principal Decision is never generated by AI.
7. Only an authorised human may record a Principal Decision.
8. At most one active Recommendation may exist at a time.
9. A Case cannot enter `awaiting_decision` without an active Recommendation.
10. Reopening a Case preserves all prior history.

## Failure Behaviour

If a Command violates a transition rule:
- the state must remain unchanged;
- no lifecycle Event must be emitted;
- the failure must return a structured domain error.

Suggested error codes:
- `INVALID_STATE_TRANSITION`
- `MISSING_OBJECTIVE`
- `MISSING_ACTIVE_RECOMMENDATION`
- `UNAUTHORISED_DECISION_MAKER`
- `CASE_ALREADY_CLOSED`
- `REOPEN_REASON_REQUIRED`
- `INVALID_CLOSURE_DATA`

## Implementation Note

The lifecycle belongs to the Case Aggregate.

External services may request transitions, but only the Aggregate may:
- validate the current state;
- enforce invariants;
- apply the transition;
- emit the resulting Event.
